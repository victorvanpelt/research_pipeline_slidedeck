% ── Horizontal process flow ────────────────────────────────────
% Usage: \process{Step 1,Step 2,Step 3,Step 4}
% Usage: \process{Step 1/whu_lightblue,Step 2,Step 3}
\newcommand{\process}[1]{%
  \begin{center}
  \begin{tikzpicture}[
    node distance=0.3cm,
    flowstep/.style={rectangle,rounded corners=3pt,fill=whu_blue,
                 text=white,font=\small\bfseries,
                 minimum height=0.9cm,minimum width=2.4cm,
                 text width=2.2cm,align=center},
    arr/.style={-{Stealth[length=5pt]},line width=1.2pt,whu_lightblue}
  ]
    \foreach \s/\c [count=\i] in {#1} {
      \edef\nodecolor{\c}%
      \edef\nodelabel{\s}%
      \ifx\nodecolor\nodelabel\def\nodecolor{whu_blue}\fi%
      \ifnum\i=1
        \node[flowstep,fill=\nodecolor] (s\i) {\s};
      \else
        \pgfmathtruncatemacro{\prev}{\i - 1}
        \node[flowstep,fill=\nodecolor,right=of s\prev] (s\i) {\s};
        \draw[arr] (s\prev) -- (s\i);
      \fi
    }
  \end{tikzpicture}
  \end{center}%
}

% ── Circular process cycle ─────────────────────────────────────
% Usage: \processcycle{Step 1,Step 2,Step 3,Step 4}
\newcommand{\processcycle}[1]{%
  \xdef\totalsteps{0}%
  \foreach \s/\c in {#1} {%
    \pgfmathtruncatemacro{\tmp}{\totalsteps+1}%
    \xdef\totalsteps{\tmp}%
  }%
  \begin{center}
  \begin{tikzpicture}[
    flowstep/.style={rectangle,rounded corners=3pt,fill=whu_blue,
                 text=white,font=\small\bfseries,
                 minimum height=0.9cm,minimum width=2.4cm,
                 text width=2.2cm,align=center},
    arr/.style={-{Stealth[length=5pt]},line width=1.2pt,whu_lightblue}
  ]
    \foreach \s/\c [count=\i] in {#1} {%
      \edef\nodecolor{\c}%
      \edef\nodelabel{\s}%
      \ifx\nodecolor\nodelabel\def\nodecolor{whu_blue}\fi%
      \pgfmathsetmacro{\angle}{90 - (\i-1)*360/\totalsteps}%
      \node[flowstep,fill=\nodecolor] (s\i) at (\angle:3cm) {\s};%
    }
    \foreach \s/\c [count=\i] in {#1} {%
      \pgfmathtruncatemacro{\next}{int(mod(\i,\totalsteps)+1)}%
      \draw[arr] (s\i) -- (s\next);%
    }
  \end{tikzpicture}
  \end{center}%
}

\begin{frame}{But there is more at stake!}
\phantomsection\label{but-there-is-more-at-stake}
\centering

In 2019, \emph{Management Science} introduced a Data and Code Disclosure
Policy:

\emph{``Authors of accepted papers\ldots{} must provide\ldots{} the
data, programs, and other details of the experiment and computations
sufficient to permit replication.''}

A 2023 study examined nearly 500 articles published under this policy to
see if the results could, in fact, be reproduced (Fišar et al. 2023).

\textbf{What share of these articles do you think were largely
reproducible?}
\end{frame}

\begin{frame}{1. Directory Structure: Fundamental Principles}
\phantomsection\label{directory-structure-fundamental-principles}
\centering

Following principles is critical, especially for projects with lots of
data and code:

\begin{itemize}
\tightlist
\item
  Anyone can run the code anywhere and anytime, regardless of location.
\item
  Anyone can understand the code and data without much effort.
\item
  Anyone can update the code on the spot without breaking it (dynamic
  coding).
\end{itemize}
\end{frame}

\begin{frame}{1. Directory Structure: How to accomplish this?}
\phantomsection\label{directory-structure-how-to-accomplish-this}
\begin{itemize}
\tightlist
\item
  Use a flow-inspired folder structure:

  \begin{itemize}
  \tightlist
  \item
    E.g., formatting files -\textgreater{} generating variables
    -\textgreater{} conducting analyses -\textgreater{} generating
    output.
  \end{itemize}
\item
  Leave the raw data untouched: You load it and use it to produce
  output:

  \begin{itemize}
  \tightlist
  \item
    E.g., raw data files -\textgreater{} input files -\textgreater{}
    process files -\textgreater{} output files.
  \end{itemize}
\item
  Use relative paths (e.g., ``\emph{../0\_data/data.csv}'') and not
  direct paths (e.g.,
  ``\emph{C:/user{[}name{]}/Documents/research/project\_1/0\_data/data.csv}'').
\item
  The code is deterministic. Random and stochastic processes ideally
  require a seed.
\item
  Do not leave blindspots in your coding and analysis

  \begin{itemize}
  \tightlist
  \item
    Use plenty of comments to explain what the code is doing.
  \item
    Be as descriptive as you can in folder and filenames
  \end{itemize}
\item
  Use environmental variables for credentials and software paths
\end{itemize}
\end{frame}

\section{4. AI Agents and Your
Pipeline}\label{ai-agents-and-your-pipeline}

\begin{frame}{4. AI agents: Opening question}
\phantomsection\label{ai-agents-opening-question}
An AI hands you a beautiful regression table. Can you defend every
number in it?

\begin{enumerate}
\item
  Yes, I built and reran every step
\item
  Probably, I would need to retrace some steps
\item
  No, I could not say where it came from
\end{enumerate}

\note{Picture an agent that just ran a robustness check and handed back
a polished regression table, formatted, significant stars included. It
looks publication ready. The real question is whether we could stand in
front of a committee and defend every number in it: where the data came
from, what transformation produced each column, why the standard errors
are clustered that way. Most of us have used a chatbot to draft or check
code at some point. Far fewer of us could immediately defend a result a
chat window produced on its own. This chapter asks how we get real help
from AI agents without losing ownership of our own results.}
\end{frame}

\begin{frame}{4. AI agents: From chatbots to agents that work inside
your project}
\phantomsection\label{ai-agents-from-chatbots-to-agents-that-work-inside-your-project}
\begin{itemize}
\tightlist
\item
  Chat window: you paste, it answers, you paste back
\item
  Agent: reads your files, edits code directly
\item
  Runs commands, reads the errors itself
\item
  Iterates until the pipeline works
\item
  Examples: Claude Code, OpenAI Codex, Copilot, Cursor
\end{itemize}

\process{Read files,Edit code,Run make,Read errors,Fix}

\note{A chat window is just a text box: we paste in code or a question,
the model answers, and we paste the answer back into our files by hand.
Nothing the model does touches the project directly. An agentic workflow
works differently. Think of cooking: instead of texting a friend for a
recipe fix, we hand them the kitchen. They open the fridge, see what's
actually there, adjust the dish, taste it, adjust again. An agent like
Claude Code, OpenAI Codex, GitHub Copilot, or Cursor reads our actual
files, edits code in place, runs a command such as make, reads the error
that comes back, and revises until it works. That loop, read, edit, run,
read errors, fix, is what turns a chatbot into a collaborator inside the
pipeline.}
\end{frame}

\begin{frame}{4. AI agents: Working in the dark makes results
irreproducible}
\phantomsection\label{ai-agents-working-in-the-dark-makes-results-irreproducible}
\begin{itemize}
\tightlist
\item
  Numbers copy-pasted from a chat window
\item
  Edits made outside git, never tracked
\item
  Prompts nobody saved or can rerun
\item
  Provenance nobody can explain later
\item
  Roughly one in three articles not reproducible, even with disclosure
  (Fišar et al. 2023)
\item
  Your committee and journals will ask you, not the AI
\end{itemize}

\note{Dark use means AI is helping, but none of that help lives in the
project. A result gets copied from a chat window straight into a draft.
An agent edits a file and nobody looks at what changed. The prompt that
produced a number sits in a chat history nobody saved. Months later,
nobody, including us, can explain where that number came from. Fišar et
al.~2023 found that roughly a third of Management Science articles could
not be reproduced, even under a strict data and code disclosure policy,
before agents were part of the picture. Add an unrecorded chat on top of
that, and reproducibility gets worse, not better. When a committee or a
journal asks us to defend a number, ``the AI produced it'' is not an
answer they will accept, and it should not be one we accept either.}
\end{frame}

\begin{frame}{4. AI agents: Your pipeline is the guardrail that keeps AI
transparent}
\phantomsection\label{ai-agents-your-pipeline-is-the-guardrail-that-keeps-ai-transparent}
\begin{itemize}
\tightlist
\item
  Numbered folders scope what the agent may touch
\item
  ``\emph{0\_data}'' stays read-only, always
\item
  git turns every agent edit into a diff
\item
  make is the contract that verifies results
\item
  README gives the agent project context
\end{itemize}

\takeaway{Your pipeline does not need to trust the agent, only to verify what it changed.}

\note{The same structure from chapters one through three is what makes
agents safe to use. Numbered folders already scope what belongs where,
so an agent editing 1\_code has no business touching 3\_output by hand.
0\_data stays read-only for the agent exactly as it does for us, real
data is never regenerated or overwritten. Because everything lives in
git, any change the agent makes shows up as a diff we can read before
committing it. make is the contract: if the pipeline runs clean from raw
data to draft, the result is verified, not just plausible. A README
gives the agent the same onboarding a new collaborator would need. None
of this requires trusting the agent's judgment, it requires trusting the
guardrails we already built.}
\end{frame}

\begin{frame}{4. AI agents: The agent edits, you review the diff}
\phantomsection\label{ai-agents-the-agent-edits-you-review-the-diff}
\begin{columns}[T]
\begin{column}{0.42\linewidth}
\begin{itemize}
\tightlist
\item
  Prompt: winsorize scaled earnings at 1st/99th percentile
\item
  Agent edits ``\emph{1\_code/code.r}'' directly
\item
  You run git diff before committing
\item
  Check: only ``\emph{1\_code}'' touched
\item
  Check: winsorize sits before the regression
\item
  Check: ``\emph{0\_data}'' and ``\emph{3\_output}'' untouched by hand
\end{itemize}
\end{column}

\begin{column}{0.58\linewidth}
\pandocbounded{\includegraphics[keepaspectratio]{images/paste-20.png}}
\end{column}
\end{columns}

\note{Say we ask an agent to add a robustness check: winsorize scaled
earnings at the first and ninety-ninth percentile before the main
regression. The agent opens 1\_code/code.r, adds the step, and reports
back that it's done. Before committing anything, we run git diff and
actually read it. Three things matter. First, scope: does the diff touch
only 1\_code, or did the agent also poke at 0\_data or 3\_output? It
should be 1\_code only, everything else gets derived by rerunning the
pipeline, never hand-edited. Second, placement: is the winsorizing step
inserted before the regression that uses it, not bolted onto the end of
the file? Third, is anything changed by hand outside the code, a number
pasted into a table, a figure edited directly? If the diff is clean on
all three, we commit it with a clear message. If not, we send the agent
back with exactly what's wrong, the same way we'd redirect a coauthor.}
\end{frame}

\begin{frame}{4. AI agents: The agent runs make, you rerun it yourself}
\phantomsection\label{ai-agents-the-agent-runs-make-you-rerun-it-yourself}
\begin{itemize}
\tightlist
\item
  Agent runs ``\emph{make -f makefile.mak}''
\item
  Reads the error, proposes a fix
\item
  Your rule: ``\emph{make clean}'', then ``\emph{make}''
\item
  Rerun it yourself, on your machine
\item
  Numbers flow from the pipeline, not a chat
\end{itemize}

\note{Suppose make -f makefile.mak fails after the agent's edit, maybe a
data type mismatch in the new winsorizing step. A good agent reads that
error itself and proposes a fix, the same way it read the file in the
first place. Letting it iterate is fine. What matters is what we do
next. Before trusting any number in the draft, we run make clean, wiping
every intermediate and output file, then make -f makefile.mak from
scratch, on our own machine, not the agent's sandbox. If the pipeline
reproduces cleanly from 0\_data through 4\_drafts, the numbers are real.
This is the payoff from chapter three: because the paper and slides are
dynamic documents pulling numbers straight from 3\_output, rerunning the
pipeline re-verifies every number in the draft. None of it comes from a
chat window, all of it comes from code we can rerun.}
\end{frame}

\begin{frame}{4. AI agents: Where could an agent silently break your
project?}
\phantomsection\label{ai-agents-where-could-an-agent-silently-break-your-project}
\discussionprompt{Where could an AI agent silently break your research project? Think alone (2 min), compare with your neighbor (2 min), share one risk with the room (1 min).}

\begin{itemize}
\tightlist
\item
  Think (2 min): where could an agent break things unnoticed?
\item
  Pair (2 min): compare notes, agree on top risk
\item
  Share (1 min): one risk per pair, aloud
\item

\end{itemize}

\note{For the next five minutes, work through this in three stages.
First, two minutes alone: where in our own workflow could an agent make

this repo. Second, two minutes with a partner: compare what we each came
up with, and agree on the one risk most likely to bite us. Third, one
minute where a few pairs share their top risk with the room. I'm
listening for a specific set of answers. Strong answers name specific
failure modes. An agent edits a file outside git, so the change never
shows up in a diff. An agent touches 0\_data directly instead of
treating it as read-only. A number lands in the draft without anyone
rerunning make. Confidential data gets pasted into a chat window and
leaves the machine entirely. If the room goes quiet during the pair
stage, here's a prompt to restart the conversation: what would the agent
have to touch for us not to notice? That question tends to surface blind
spots fast, because it forces us to think about what we are not
currently checking. We'll spend the last minute naming these out loud,
because the goal is not a perfect list, it's noticing that these failure
modes are all about visibility, not about the AI itself.}
\end{frame}

\begin{frame}{4. AI agents: Rules of engagement for reproducible AI use}
\phantomsection\label{ai-agents-rules-of-engagement-for-reproducible-ai-use}
\begin{itemize}
\tightlist
\item
  Opaque AI use breaks reproducibility
\item
  Your pipeline is the agent's guardrail
\item
  Review the diff, rerun make, every time
\end{itemize}

\small

\begin{columns}[T]
\begin{column}{0.55\linewidth}
\textbf{Do}

\begin{itemize}
\tightlist
\item
  Agent works inside the repo
\item
  Raw data in ``\emph{0\_data}'' stays read-only
\item
  Review every diff before committing
\item
  Rerun the pipeline before trusting a number
\item
  Credentials live in environment variables
\end{itemize}
\end{column}

\begin{column}{0.45\linewidth}
\textbf{Never}

\begin{itemize}
\tightlist
\item
  Chat-window analyses copied into the draft
\item
  Confidential data pasted into a chat
\end{itemize}
\end{column}
\end{columns}

\normalsize

\takeaway{AI amplifies your data management. The pipeline turns it from a black box into a reviewable collaborator.}

\note{Three things to take home. First, AI used in the dark, chat
windows and untracked edits, produces numbers nobody can defend. Second,
the pipeline we built today is exactly what keeps an agent transparent:
it works in files, under git, against make. Third, the routine is short.
We read the diff before we commit, and we rerun the pipeline before we
trust a number. Two honest caveats. Output is often wrong, confidently
wrong, so verification is not optional. And skipping AI entirely remains
a valid way to do a PhD. When the agent added our winsorizing step, the
diff showed one changed file in 1\_code, and a clean make rebuilt the
same table in the draft. That is what owning an AI-assisted result looks
like.}
\end{frame}

\begin{frame}{Summary and Wrap-up}

\begin{itemize}
\tightlist
\item
  Across all fields of science, reproducibility has been under threat
  (Open Science Collaboration 2015; Baker 2016; Camerer et al. 2016;
  Hail, Lang, and Leuz 2020; Fišar et al. 2023)
\item
  Good data management is vital to ensure results are reproducible.
\item
  You, as a WHU doctoral student, are vital:

  \begin{enumerate}
  \tightlist
  \item
    Maintain a good directory structure
  \item
    Use version control
  \item
    Integrate multiple languages (try Quarto)
  \item
    Let AI agents work inside your pipeline, never in the dark
  \end{enumerate}
\item
  If you don't want to do it for science, then do it to save yourself
  lots of time and headache.
\end{itemize}
